## What's shipped on master

### PR #595 — Pipeline hardening (`5f0f1b2`)

Three critical bugs were silently breaking the "body + comments reach the LLM" feature:

1. **Wiring bug**: `process-quick-note.yml` set `ISSUE_BODY_FILE` and `ISSUE_COMMENTS_FILE` as env vars but never passed them as `--body-file` / `--comments-file` flags to `implement_agent.py`. LLM was getting empty context. **Fixed**: added the flags to the command line.

2. **JSONL validation bug**: The capture step's non-array guard used `jq -e 'all(.[]; type == "object")'` which is wrong for JSONL (plain `jq` only reads the first top-level value). **Fixed**: per-line validation via `while IFS= read -r line; do echo "$line" | jq -e 'type == "object"' >/dev/null 2>&1; done`.

3. **Context-window bug**: `implement_agent.py` had no upper bound on the body+comments block. A 500-comment issue produced ~250K chars, exceeding every model's window. **Fixed**: `THREAD_CAP = 60_000` with tail-truncation (head=15K, tail=45K) preserving the most recent comments and a notice to the LLM to read `/tmp/issue_comments.jsonl` directly via the bash tool.

Additional hardening (from earlier rounds, all preserved):
- `process-quick-note.yml` picker: widened from `gh issue list --search "quick-note: in:title"` to `--json number,labels` + jq that drops only `quick-note:exhausted`. Oldest OPEN issue is picked regardless of title or label.
- Capture step body fetch: wrapped so a transient `gh issue view` failure does not kill the step.
- Capture step pagination: breaks early on empty page and at page 5.
- Fetch step: no longer exits 1 on empty URL; writes a stub using the issue body.
- `issue-context-generator.yml`: `if:` condition is `!contains(...'automation-test') && (action == 'opened' || (action == 'labeled' && label.name == 'quick-note'))`.

### PR #599 — Draft-PR safety (`817def2`)

Closed the gap where a draft PR (e.g., issue-context-generator-created PR containing only a planning doc) could be auto-merged by `auto-merge.yml` the moment CI passed.

1. **`.github/workflows/auto-merge.yml`**: pre-merge guard checks `isDraft` via `gh pr view --json isDraft` and skips the merge if true. Also checks for the `quick-note:planning` label as belt-and-suspenders.

2. **`.github/workflows/process-quick-note.yml`**: split the merge step into two:
   - New "Mark PR ready for review (draft → ready)" step runs **first**. It calls `gh pr ready` if the PR is still draft, then removes the `quick-note:planning` label.
   - The merge step's `if:` now requires `steps.ready.outputs.ready == 'true'`. It also has a belt-and-suspenders `isDraft` check in the `run:` block that exits 1 if still draft (catches silent failures from mark-ready).

3. **`.github/workflows/issue-context-generator.yml`**: applies the `quick-note:planning` label to draft PRs as a fast-path filter for future housekeeping workflows. Comment explains the intentional `|| true` (canonical guard is the `isDraft` check).

4. **`.github/scripts/implement_agent.py`**: "DRAFT PR AWARENESS" line in the system prompt — agents know not to signal `IMPLEMENTATION_COMPLETE` on a draft PR because it is "yet to be implemented" and needs real code commits.

### PR #600 — ASCII cleanup (`2839eec`)

One-line cosmetic: replaced Unicode em-dash (U+2014) in the DRAFT PR AWARENESS line with ASCII `--`. Functional hardening unchanged.

### P3 — Tighten `$ISSUE_TASK` shell-quoting in `implement_agent.py`

**Why**: `process-quick-note.yml` invokes `implement_agent.py` with positional `url`, `issue_num`, `task` and only adds `--body-file` / `--comments-file` as named flags. `$ISSUE_TASK` can contain spaces (it's the `Task:` line of the issue body, frequently multi-word). argparse `nargs="?"` will absorb `--something` into the task string if a future prompt format puts the flags first. Cheap robustness fix.

**What to do**: In `implement_agent.py`, switch to named flags `--task`, `--issue-num`, `--url` and deprecate the positional ones. Update `process-quick-note.yml` accordingly. The argparse already uses `parse_known_args()` so this is backward-compatible — old positional invocations still work.
